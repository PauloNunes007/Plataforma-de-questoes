# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Questly — a gamified study-planner web app (Portuguese/pt-BR UI) for university students prepping for exams ("Bosses"). Static HTML/CSS/JS front end talking directly to Supabase (Postgres + Auth + RLS) from the browser. No backend server, no build step, no bundler, no package.json, no test suite. Every dependency is loaded from a CDN `<script>` tag in each HTML file.

## Running / developing

There is no build or lint command — open the HTML files directly in a browser, or serve the folder with any static file server (e.g. `npx serve .`) since Supabase auth redirects work better over `http://` than `file://`. To change the backend schema, run the `supabase_*.sql` files by hand in the Supabase SQL Editor, in the order noted in each file's header comment (they're additive, idempotent migrations — not a managed migration tool). `supabase_conteudo_compartilhado.sql` must run before `supabase_seed_questoes.sql`. `supabase_ementas_ciclo_basico.sql` (curricular content + skip/recap columns + basic-engineering-cycle ementas) runs after `supabase_conteudo_compartilhado.sql`; it renames the `Geometria Analítica` materia to `Fundamentos de Cálculo e Geometria` and `Programação` to `Programação I`, and adds a unique index on `topicos (materia_id, lower(nome))` (its seeds upsert on that key). `supabase_importar_revisao.sql` runs after `supabase_add_storage_imagens.sql`; it adds `questions.alternativas_imagens` (per-letter image URLs, used by the JSON reviewer importer). `supabase_ciencia_aprendizagem.sql` runs after `supabase_conteudo_compartilhado.sql`; it adds `question_attempts.motivo_erro` (check-constrained to `conceito|calculo|interpretacao|chute`) plus an owner-only UPDATE policy on `question_attempts` — without that policy the error-autopsy update in `questao.js` silently updates 0 rows (RLS filters, no error). `supabase_rotina_semanal.sql` runs after `supabase_conteudo_compartilhado.sql`; it adds the `rotina_semanal` table (owner-only RLS) that backs the weekly per-discipline schedule (see "Weekly schedule & multi-mission days" below).

## Page → script map

Every protected page loads scripts in this order: `@supabase/supabase-js@2` (CDN) → `js/supabase-client.js` → page-specific script(s).

- `questly_landing.html` — marketing page, no Supabase.
- `questly_login.html` — login/signup, uses `supabase-client.js` only.
- `questly_onboarding.html` — setup wizard (profile + disciplinas + provas), inlines its own save logic (`salvarCampanha`) rather than a separate JS file.
- `questly_dashboard.html` → `js/rotina-engine.js` + `js/mission-engine.js` + `js/liga.js` + `js/dashboard.js` — home screen: greeting/XP, disciplina list, a mission card **per discipline scheduled today** (see "Weekly schedule & multi-mission days"), the Duolingo-style "trilha" path, boss card, league card, streak heatmap, calendar.
- `questly_disciplinas.html` → `js/disciplinas.js` — free practice: pick disciplina/tópicos/dificuldade/quantidade, creates a `missions` row with `avulsa=true` and hands off to `questao.html` (same XP/coverage effects as the daily mission; the dashboard's daily-mission lookup filters `avulsa=false`).
- `questly_trilha.html` → `js/trilha.js` — "Minha trilha": per-discipline syllabus (the ementa in curricular order) showing each topic's status; lets a student who joined mid-semester mark topics they already know — **Já sei** (skip → `aluno_topico_progresso.status='pulado'`, no XP, drops out of daily missions) or **Fazer recap** (a short `avulsa` mission tagged `recap_topico_id`; scoring ≥70% sets `status='dominado'`). Reachable from the "Minha trilha" 🧭 nav link now present in every sidebar.
- `questly_ranking.html` → `js/liga.js` + `js/ranking.js` — weekly league ranking; clicking a row opens a public student card (photo, name, curso, disciplinas).
- `questao.html` → `js/chance-aprovacao.js` + `js/liga.js` + `js/questao.js` — answering a mission's questions (KaTeX-rendered, optional `imagem_url` figure and per-alternative `alternativas_imagens[letra]` figures), plus loads KaTeX from CDN.
- `questly_configuracoes.html` → `js/rotina-engine.js` + `js/configuracoes.js` — edit routine/disciplinas/provas after onboarding, the weekly schedule grid (discipline × day checkboxes, with a "recommend" button — see below), plus profile-photo upload (browser-side resize to 256×256 JPEG → Storage bucket `avatars/{uid}/avatar.jpg` → public URL with `?v=` cache-bust saved to `profiles.foto_url`).
- `questly_importar.html` → `js/importar.js` — content-authoring tool: paste/upload a JSON array of raw questions (each item can carry `materia`/`topico` by name and `imagem_enunciado`/`alternativas_com_imagem` flags). `avaliarElegibilidadeAuto` splits the batch into two paths: items with **no pending image flag** and **no structural inconsistency** (materia/topico resolve against `materias`/`topicos` by name, gabarito matches a filled alternative, dificuldade valid, LaTeX `$`/`{}` roughly balanced, no duplicate — see below) get bulk-inserted straight into `questions` with no per-item UI (`importarAutomaticamente`, batched like the old CSV importer); everything else — has an image or fails a check — goes to the one-by-one review queue, a card that mirrors `questao.html`'s exact visual (live KaTeX preview, `throwOnError:false` so bad LaTeX renders inline instead of breaking the card) with editable fields for every column, image upload/URL/**paste-from-clipboard** pickers for the enunciado figure and each alternative (uploads go to the `questoes` Storage bucket, auto-compressed to JPEG client-side; alternative text is optional when an image is supplied — `letraAtiva` treats text-or-image as "filled"), and a red banner explaining why that item needs a human (`atualizarMotivoBanner`). Duplicate detection (`enunciadosExistentes`, a Set of every existing `questions.enunciado` normalized to lowercase/collapsed-whitespace, fetched once on load and kept updated as items get inserted this session) catches an enunciado matching something already in the DB or repeated within the same uploaded file; it's exact-normalized-text matching, not fuzzy, and approving a flagged duplicate anyway requires an extra confirm(). This automatic check is structural/consistency only, not proofreading — matéria/tópico dropdowns are sourced from the existing `materias`/`topicos` tables and are never created here, only picked/corrected. Approving inserts (or updates, if revisited) straight into `questions`; skipping just advances the queue. The in-progress queue autosaves to `localStorage` (`questly_importar_fila_v1`) so a page refresh mid-batch doesn't lose work. See the file header comment for the exact expected JSON shape per question.
- `listas_questoes/` — question-bank source files (CSV + PDF), reference material for hand-authoring the JSON fed into the importer; not app code.

## Auth pattern

`js/supabase-client.js` creates the single global client `supabaseClient` (named to avoid colliding with the CDN lib's own `supabase` global) and exposes reusable `questly*`-prefixed helpers used across pages: `questlyCadastrar`, `questlyLogin`, `questlyLogout`, and `questlyExigirLogin` (route guard — call at the top of every protected page's `DOMContentLoaded` handler; redirects to login if there's no session and lazily creates the `profiles` row via `questlyGarantirProfile` if missing, which covers the email-confirmation signup path where no profile exists yet).

## Data model — shared content vs. per-student progress

This split is the main thing to understand before touching data logic, since it spans several files:

- **Shared/global** (not scoped to a user): `materias` (subject taxonomy, e.g. "Cálculo II"), `topicos` (topics under a materia, flagged `cai_na_prova`, plus `ordem` = position in the materia's ementa and `descricao` = the subtopics text; `ordem` drives the curricular order missions follow — a topic with no `ordem` sorts to the end), `questions` (belong to a `topico`, `alternativas` is a JSONB object keyed by letter (text only), `alternativas_imagens` is an optional JSONB object keyed by letter (image URL, only letters that have one), `gabarito` is the correct letter, `tempo_medio_seg` is a running estimate, optional `imagem_url` enunciado figure). Any authenticated user can read/write this content (RLS has no admin role yet — same trust level as before, see `supabase_conteudo_compartilhado.sql`).
- **Per-student**: `profiles` (name, curso, xp_total, nivel, streak_atual, dias_disponiveis, tempo_diario_min — now up to 480 (8h+), see "Weekly schedule & multi-mission days" — plus league state: `liga`, `xp_semana`, `questoes_semana`, `semana_inicio`, `foto_url`), `subjects` (a student's personal enrollment in a materia, via `materia_id`; holds `nota_desejada` and the computed `chance_aprovacao`), `bosses` (a subject's exam dates + computed `preparo_percentual`), `campaigns` (subject↔student link created alongside a subject), `rotina_semanal` (which weekday(s) each subject is scheduled on — `subject_id` + `dia_semana`, unique per user/subject/day; see below), `missions` (a generated daily plan, now one row per subject scheduled that day: `topic_ids`, `question_ids`, `qtd_questoes`, `tempo_previsto_min`, `xp_recompensa`, `concluida`, `avulsa`, `recap_topico_id` — set only on recap missions from the trilha page), `question_attempts` (answer log, plus `motivo_erro`), `aluno_topico_progresso` (per-student per-topic `taxa_acerto`/`num_questoes_respondidas`/`ultima_revisao` and `status` = `pendente`|`pulado`|`dominado` — progress lives here, not on the shared `topico`), `daily_logs` (per-day study flag, drives streak/heatmap/calendar), `historico_semanal` (closed-week league snapshots).
- **RLS visibility**: `profiles`, `subjects`, and `historico_semanal` are readable by any authenticated user (the ranking and public student card need them); writes stay owner-only. `questions` has an authenticated UPDATE policy — without it the `tempo_medio_seg` recalibration in `questao.js` silently updates 0 rows (RLS filters, no error).
- **Storage** (Supabase Storage, free tier): public buckets `avatars` (per-user folder `{uid}/avatar.jpg`, owner-only writes) and `questoes` (question figures — enunciado + per-alternative — authenticated uploads; `js/importar.js` uploads directly from the review card, client-side resized/compressed to JPEG, but a plain public image URL works too).

## Weekly schedule & multi-mission days (`js/rotina-engine.js`)

The student picks **which disciplines to study on which weekday** — the app no longer silently studies just one subject/day. This is the `rotina_semanal` table (`supabase_rotina_semanal.sql`): one row per `(user_id, subject_id, dia_semana)`, `dia_semana` in the same 3-letter lowercase format as `QUESTLY_DIAS_SEMANA` (`js/supabase-client.js`, index = `Date.getDay()`). Edited on `questly_configuracoes.html` ("Grade semanal" card — a discipline × day checkbox grid); `questly_onboarding.html` seeds a simple default (every discipline on every chosen `dias_disponiveis` day) at signup so the app is usable immediately, before there's `chance_aprovacao` data to make a smarter call.

`js/rotina-engine.js` provides the math, always editable/overridable by the student — it's a recommendation, not a constraint:
- `questlyPesoDisciplina(subject, hoje)` — a weight per subject combining exam urgency (`1/(diasAteProva+3)`, boss-less subjects default to a 60-day equivalent), weakness (`1 - chance_aprovacao/100`, defaults to neutral 0.5 with no data yet), and goal (`nota_desejada/10`), blended with the same `PESO_*` (0.45/0.35/0.20) convention as `chance-aprovacao.js`.
- `questlyRecomendarRotina(subjects, diasDisponiveis, tempoDiarioMin)` — recommends which days each subject lands on via a **weighted credit scheduler** (Weighted Fair Queuing / Deficit Round Robin, the same principle OS/network schedulers use to divide a resource by weight without starving or monopolizing): each day, every subject's credit grows by its weight; the top `questlyDisciplinasPorDia` subjects by credit get that day and have their credit debited by the mean weight. Over a week, each subject's frequency converges to its weight's proportion. `questlyDisciplinasPorDia` caps how many subjects fit a day at `tempo_diario_min / QUESTLY_MIN_MINUTOS_POR_DISCIPLINA` (75min), capped again at `QUESTLY_MAX_DISCIPLINAS_POR_DIA` (4) — that cap only shapes the *recommendation*; manual edits in Configurações aren't limited.
- `questlyApportionarMinutos(subjectsHoje, tempoDiarioMin)` — splits a single day's `tempo_diario_min` across that day's scheduled subjects proportional to the same weight, via the largest-remainder (Hamilton apportionment) method — used by `mission-engine.js` to size each subject's mission.
- `questlyBuscarRotinaCompleta`/`questlySalvarRotina` — read/full-replace the student's `rotina_semanal` rows (delete-then-insert, same pattern as overwriting `profiles.dias_disponiveis`).

## Mission generation (`js/mission-engine.js`)

`questlyGerarMissoesDoDia(user, profile)` runs on every dashboard load and is idempotent — it returns today's already-existing missions plus generates any still missing, so it's safe to call every time instead of only "if none exists":
1. Respects `profile.dias_disponiveis` — returns `{ missoes: [], semMissaoHoje: true, motivo }` on non-study days.
2. Reads the weekly schedule (`questlyBuscarRotinaCompleta`). **No `rotina_semanal` rows at all** (pre-existing accounts from before this feature) falls back to the old single-subject behavior (nearest-boss subject only). With a schedule configured but nothing checked for today, that's an intentional rest day for that day — no mission is invented, motivo points at Configurações.
3. `profile.tempo_diario_min` is split across today's scheduled subjects via `questlyApportionarMinutos` (weighted, not even) — a subject with a closer exam or lower `chance_aprovacao` gets more of the day's time, not just more days/week.
4. For each scheduled subject that doesn't already have a mission today, `questlyGerarMissaoParaSubject(user, profile, subject, tempoAlocadoMin)` runs the actual per-subject algorithm (unchanged from before this feature, just parameterized by subject + its allocated minutes instead of always "the boss subject" + the full daily budget):
   - **Curricular frontier**: only topics that (a) have questions in the bank and (b) aren't `pulado`/`dominado` are eligible. Among the eligible, the *frontier* is the first-by-`ordem` topic not yet covered (`num_questoes_respondidas < COBERTURA_TOPICO_QUESTOES`, currently 5). The mission draws only from the frontier + earlier topics (revision) — **it never jumps ahead in the ementa**. Topics with no questions don't block the frontier (ementas vary by university; the frontier steps over them). When everything is covered, it switches to pure-revision mode over all eligible topics.
   - Eligible/frontier-scoped topics are scored by: a `BONUS_FRONTEIRA` so the new frontier topic almost always makes the cut, `cai_na_prova` weight, low `taxa_acerto` (weak content), Ebbinghaus retention (see "Learning science" below, with `BONUS_REVISAO_URGENTE` for overdue topics), and extra weight for weak topics when `nota_desejada >= 9`.
   - Top `MAX_TOPICOS_POR_MISSAO` (5 — raised from the original 3 so a subject with hours allocated doesn't run dry) topics are kept; candidate questions are shuffled, overdue-review and frontier questions front-loaded, then greedily added until the *allocated* minutes are filled (respecting `MIN_QUESTOES`/`MAX_QUESTOES`, the latter raised to 40 — the original ceiling of 15 was tuned for a single ~30min/day budget and felt shallow once a student can allocate a 2h+ block to one subject).
   - The exact `question_ids` are fixed on the mission row at creation time (not just `topic_ids`) so the displayed time estimate is a real sum, not a re-sampled average — `questao.js` prefers `question_ids` and only falls back to querying by `topic_ids` for missions created before this existed.
5. A subject with no eligible topics/questions yet is skipped without blocking the other scheduled subjects' missions that day.

## XP economy

XP per question is difficulty-weighted via `questlyXpDaQuestao` in `supabase-client.js` (`facil: 3, medio: 5, dificil: 8`, fallback 5) — mission `xp_recompensa` is the exact sum over the chosen questions, and `questao.js` pays the same weights per correct answer. A question the student already answered correctly in a past attempt pays half XP (anti-farming; `questao.js` preloads the set from `question_attempts`). Finishing a mission bumps `xp_total` and the weekly `xp_semana`/`questoes_semana` counters together (`atualizarXpELiga`).

## Weekly ranking & leagues (`js/liga.js`, `js/ranking.js`)

Five leagues (bronze→diamante), weekly rounds keyed by the local-time Monday (`questlySegundaDaSemana`). There is no server cron: rollover is lazy — `questlyGarantirSemanaLiga` runs wherever league state is needed and, if the profile's `semana_inicio` is stale, snapshots the closing week into `historico_semanal`, computes the fate, and resets the counters. Promotion/demotion is decided by the pure function `questlyDestinoNaLiga` (the ranking page paints its zones with the same function, so display always matches outcome). Rules: competition ranking on strictly-greater XP counts (deterministic ties — tied-at-top all promote; a tied group only demotes if the whole group fits in the zone), promotion requires `xp_semana > 0`, demotion only exists when the group has ≥ 5 members (small-sample protection), quotas are 30%/30%. When touching this math, keep the function pure and deterministic — every client must independently reach the same verdict.

## Approval-chance heuristic (`js/chance-aprovacao.js`)

`questlyCalcularMetricas` is a pure function (no Supabase calls — caller fetches data, this just computes) combining coverage (`num_questoes_respondidas` vs. `META_QUESTOES_TOPICO`), precision (`taxa_acerto` weighted by volume), and recent study frequency into `chanceAprovacao`. It explicitly returns `null` (not a low number) when a data-sufficiency gate isn't met (`MIN_QUESTOES_TOTAL`, `MIN_FRACAO_TOPICOS_TOCADOS`, `MIN_DIAS_COM_ESTUDO`) — the UI must show "no data yet" rather than a fabricated percentage. This is explicitly documented as a transparent heuristic, not a validated prediction — don't make it look more authoritative than that. It's invoked from `questao.js`'s `atualizarMetricasSubject` after each mission finishes, writing `bosses.preparo_percentual` (coverage only) and `subjects.chance_aprovacao` (full formula). `atualizarMetricasSubject` **excludes `status='pulado'` topics** from the topic list it passes in — a self-declared skip shouldn't count as 0% coverage dragging the chance down; `dominado` topics stay in because the recap gave them real data. The coverage threshold is duplicated as `META_QUESTOES_TOPICO` here, `COBERTURA_TOPICO_QUESTOES` in `mission-engine.js`, and `COBERTURA_TOPICO` in `trilha.js` (chance-aprovacao.js isn't loaded on the dashboard/trilha pages) — keep all three in sync.

## Skip / recap (mid-semester entry)

A student who joins after the term started shouldn't grind through content they already know. On `questly_trilha.html` each ementa topic can be **pulado** (self-declared "já sei" → `status='pulado'`, no XP, removed from daily missions and from the approval-chance denominator; reversible from the same page) or proven via a **recap**: a short single-topic `avulsa` mission with `recap_topico_id` set. `questao.js`'s `avaliarRecap` (called in `finalizarMissao`, before `atualizarMetricasSubject`) reads that field and, if the score ≥ `RECAP_APROVACAO` (0.7), sets `status='dominado'`; recaps pay normal XP because the student answered real questions. Both statuses drop the topic out of `mission-engine.js`'s eligible set, which is what lets the curricular frontier advance past already-known early topics.

## Learning science (mastery, spaced repetition, retrieval, metacognition)

Four mechanisms layered on the existing tables — no new tables, no RLS model change (see `supabase_ciencia_aprendizagem.sql`):

- **Mastery (`questlyEhMestre` in `supabase-client.js`)**: a topic is "Mestre" when `taxa_acerto >= QUESTLY_MAESTRIA_TAXA` (0.9) with `num_questoes_respondidas >= QUESTLY_MAESTRIA_MIN_QUESTOES` (20). It's *derived* from `aluno_topico_progresso`, never stored. UI: gold `🏅 Mestre` badge on the trilha, and the dashboard mission banner goes gold ("Missão de Mestre") when every `topic_ids` topic is mastered. Correct answers in a topic that was already mastered *at mission start* pay `QUESTLY_MAESTRIA_MULT_XP` (1.5×), applied after the repeat-half rule (`questao.js` snapshots `topicosMestreInicio` on load so the rate can't flip mid-mission). `questao.js` also celebrates newly-crossed masteries on the result screen (`celebrarNovasMaestrias`).
- **Spaced repetition (Ebbinghaus)**: `questlyRetencaoTopico` estimates retention `R = e^(-t/S)` where `S` (stability, days) grows with volume × accuracy (`questlyEstabilidadeDias`). In `mission-engine.js`, a covered topic with `R < QUESTLY_RETENCAO_LIMIAR` (0.6) becomes *revisão urgente*: it gets `BONUS_REVISAO_URGENTE` (45, deliberately > `BONUS_FRONTEIRA`) and its questions are front-loaded *before* the frontier's new-content dose — overdue memory beats new content. The frontier/never-jump-ahead invariant is untouched (urgent topics are always at-or-before the frontier by construction).
- **Retrieval practice**: after finishing a non-`avulsa` mission, `prepararDesafioRecuperacao` (`questao.js`) offers one random question from a topic untouched for 7+ days (`DESAFIO_DIAS_SEM_TOCAR`), scoped to the student's own materias. Accepting creates a 1-question `avulsa` mission (so it never chains into another challenge and doesn't occupy the daily slot).
- **Metacognition (error autopsy)**: on a wrong answer, `questao.js` shows optional chips writing `question_attempts.motivo_erro` (update of the just-inserted attempt row — needs the UPDATE policy from `supabase_ciencia_aprendizagem.sql`). `atualizarMetricasSubject` counts the student's classified errors for the materia and passes them as the optional 5th arg of `questlyCalcularMetricas`; `PERDAO_POR_MOTIVO` (chance-aprovacao.js) refunds a fraction of each error into an *adjusted* precision used only in the score (`calculo` 0.5, `interpretacao` 0.3, `conceito`/`chute` 0) — displayed precision stays raw, and unclassified errors keep full penalty, so the param is fully backward-compatible.

## Conventions

- All UI strings, and most function/variable names, are in Portuguese matching the app's domain vocabulary (missão, disciplina, prova/boss, trilha, cobertura, preparo). Keep new code consistent with this rather than switching to English identifiers.
- Cross-page reusable functions (in `supabase-client.js`, `mission-engine.js`, `chance-aprovacao.js`) are prefixed `questly` to avoid name collisions with page-local scripts that define similarly-named helpers (e.g. `escapeHtml`, `embaralhar` are redefined per-file rather than shared).
- Dates from Supabase (`timestamptz`) are normalized to `YYYY-MM-DD` with `String(x).slice(0, 10)` before comparing or feeding an `<input type="date">` — a raw timestamp string silently blanks a date input.
- `questions.tempo_medio_seg` is continuously recalibrated via an exponential moving average (`anterior*0.7 + novo*0.3`) as students answer, so mission-engine's time budgeting improves over time instead of relying on the CSV-import-time estimate forever.
